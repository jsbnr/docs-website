**컨텍스트에서 New Relic의 로그를** 활성화하면 로그 데이터를 APM 및 인프라 에이전트에서 수집한 데이터와 연관시킵니다. 즉, APM, 인프라 모니터링, 분산 추적 및 오류 받은 편지함과 같은 다른 UI 환경에서 로그 데이터 또는 로그 데이터에 대한 링크를 볼 수 있습니다. 로그 데이터를 수동으로 검색할 필요 없이 해당 로그 데이터를 사용하여 앱 또는 호스트의 성능을 더 깊이 파고들 수 있습니다.

컨텍스트 및 [로그 패턴](/docs/logs/ui-data/find-unusual-logs-log-patterns) 의 로그가 문제의 근본 원인을 찾는 데 어떻게 도움이 되는지 보려면 다음 짧은 비디오(약 5분)를 시청하십시오.

## 플랫폼 전반에 걸친 문제의 근본 원인 확인 [#compatibility-requirements]

모든 애플리케이션 및 인프라 데이터를 단일 솔루션으로 통합하면 문제의 근본 원인을 더 빨리 파악할 수 있습니다. 컨텍스트에 로그인하면 의미 있는 패턴과 추세를 빠르게 확인할 수 있습니다.

다음 다이어그램은 에이전트 메타데이터를 통한 강화(컨텍스트 로깅)에서 로그 데이터의 형식 지정 및 New Relic으로 전달에 이르기까지 로그 메시지의 수명 주기를 보여줍니다.

플랫폼의 다른 부분에서 모든 로그의 범위를 좁히려고 추가 시간을 소비하지 마십시오. 대신 컨텍스트에서 로그를 활성화하여 문제를 식별하고 해결하는 데 필요한 정확한 로그 라인을 확인하십시오.

## 컨텍스트에서 로그를 활성화하는 기본 프로세스 [#enable-logs]

다음은 컨텍스트에서 로그를 표시하는 New Relic APM UI의 예입니다. 선택한 기간의 이벤트 컨텍스트에서 로그를 보고 강조 표시된 속성에 대한 자세한 데이터로 드릴다운할 수 있습니다. 더 많은 기능을 활용하려면 여기에서 **로그 쿼리** 를 클릭하여 **로그** UI로 직접 이동합니다.

<img
  title="APM UI의 컨텍스트 로그인"
  alt="APM UI의 컨텍스트 로그인"
  src={demotronLogs123021}
/>

APM 및 인프라 모니터링을 위해 컨텍스트에서 로그를 설정하는 방법은 다음과 같습니다.

1. [New Relic 에서 이미 로그인을 설정](/docs/logs/enable-log-management-new-relic/enable-log-monitoring-new-relic/enable-log-management-new-relic) 했는지 확인하십시오. 여기에는 애플리케이션 로그를 수집하고 New Relic으로 전달되는 메타데이터를 확장하는 지원되는 로그 전달자 구성이 포함됩니다.
2. APM에 대한 컨텍스트에서 로그인을 원하는 경우: 먼저 지원되는 APM 에이전트 버전으로 업데이트하고 [분산 추적](/docs/distributed-tracing/concepts/introduction-distributed-tracing) 을 활성화하십시오. 특정 지침을 보려면 에이전트를 선택하세요.

* [C SDK](/docs/logs/enable-log-management-new-relic/configure-logs-context/c-sdk-configure-logs-context)
* [가다](/docs/logs/enable-log-management-new-relic/configure-logs-context/configure-logs-context-go/)
* [자바](/docs/logs/enable-log-management-new-relic/configure-logs-context/java-configure-logs-context-all/)
* [.그물](/docs/logs/enable-log-management-new-relic/configure-logs-context/net-configure-logs-context-all/)
* [노드.js](/docs/logs/enable-log-management-new-relic/configure-logs-context/configure-logs-context-nodejs/)
* [PHP](/docs/logs/enable-log-management-new-relic/configure-logs-context/configure-logs-context-php/)
* [파이썬](/docs/logs/enable-log-management-new-relic/configure-logs-context/configure-logs-context-python/)
* [루비](/docs/logs/enable-log-management-new-relic/configure-logs-context/configure-logs-context-ruby/)

3. 인프라 모니터링을 위해 컨텍스트에서 로그를 원하는 경우 [인프라 에이전트에 대해 컨텍스트에서 로그 활성화를](/docs/logs/enable-log-management-new-relic/enable-log-monitoring-new-relic/forward-your-logs-using-infrastructure-agent) 참조하십시오.
4. [모니터링되는 앱 및 호스트와 관련하여 로그 데이터를](#what-next) 확인합니다.

## API 및 기타 옵션 [#API]

당사의 로깅 솔루션이 귀하의 요구 사항을 충족하지 못하는 경우 다른 옵션을 사용하여 로그 데이터를 New Relic으로 보낼 수 있습니다.

* [에이전트 API 호출](/docs/logs/new-relic-logs/apm-integration/apm-agent-apis-manual-logs-context-instrumentation) 을 통한 로깅 확장
* [Log API](/docs/logs/log-management/log-api/introduction-log-api/) 를 통한 HTTP 엔드포인트
* [TCP 끝점](/docs/logs/log-management/log-api/use-tcp-endpoint-forward-logs-new-relic/) 을 통한 Syslog 프로토콜(CDN, 하드웨어 장치 또는 관리 서비스에 유용)

## 다음은 뭐지? [#what-next]

APM 또는 인프라 모니터링을 위해 컨텍스트에서 로그를 설정한 후 New Relic One UI에서 로깅 데이터를 최대한 활용하십시오.

* 당사의 [로그 UI](/docs/logs/log-management/ui-data/use-logs-ui) 를 사용하여 플랫폼 전반에 걸쳐 로깅 데이터를 탐색하십시오.
* [APM UI](/docs/logs/log-management/ui-data/use-logs-ui/#links) 의 앱 성능 컨텍스트에서 로그를 확인합니다. [분산 추적](/docs/apm/distributed-tracing/ui-data/understand-use-distributed-tracing-data) , 스택 추적, 애플리케이션 로그 등으로 [오류](/docs/apm/apm-ui-pages/error-analytics/errors-page-find-fix-verify-problems/) 를 해결합니다.
* [인프라 모니터링 에이전트](/docs/logs/enable-log-management-new-relic/enable-log-monitoring-new-relic/forward-your-logs-using-infrastructure-agent/) 를 통해 로그를 전달하여 애플리케이션과 플랫폼 성능 데이터에 대한 더 깊은 가시성을 얻으십시오. UI에서 [인프라 로그](/docs/logs/log-management/ui-data/use-logs-ui/#links) 를 검토합니다.
* [경보](/docs/alerts-applied-intelligence/new-relic-alerts/alert-conditions/create-alert-conditions/)를 설정하십시오.
* [데이터를 조회](/docs/query-your-data/explore-query-data/get-started/introduction-querying-new-relic-data/) 하고 [대시보드를 작성](/docs/query-your-data/explore-query-data/dashboards/introduction-dashboards/)하십시오.
